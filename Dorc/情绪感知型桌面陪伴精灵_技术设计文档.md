 # 情绪感知型桌面陪伴精灵技术设计文档

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 产品名称 | 情绪感知型桌面陪伴精灵 |
| 文档类型 | 技术设计文档 |
| 对应 PRD | 情绪感知型桌面陪伴精灵产品需求文档 V1.0 |
| 当前阶段 | MVP 技术方案 |
| 技术目标 | 快速实现可演示、可扩展、隐私友好的桌面陪伴闭环 |

## 2. 技术设计目标

本项目不是单点算法项目，也不是普通桌面工具。技术设计需要支撑一个完整产品闭环：

```text
摄像头本地感知
-> 用户状态趋势判断
-> 桌面宠物表现变化
-> 小游戏、待办、对话联动
-> 本地记录和持续陪伴
```

MVP 阶段的技术目标如下：

1. 支持桌面宠物透明悬浮、置顶显示和低打扰交互。
2. 支持本地摄像头采集和基础面部状态识别。
3. 支持基于趋势的用户状态判断，而不是单帧情绪判断。
4. 支持宠物状态、小游戏、待办、对话之间的事件联动。
5. 支持本地存储任务、设置、宠物成长数据。
6. 默认不保存、不上传原始人脸图片或视频。
7. 保持架构可扩展，后续可以升级 Live2D、更多游戏和大模型对话。

## 3. 技术选型总览

| 模块 | 最终选型 | 选型结论 |
| --- | --- | --- |
| 桌面应用框架 | Electron | MVP 首选，桌面窗口能力成熟，适合透明悬浮桌宠 |
| 前端框架 | React + TypeScript + Vite | 状态驱动 UI 清晰，开发效率高 |
| 桌宠动画 | Lottie / WebP 序列帧 | 实现成本低，视觉效果稳定 |
| 摄像头采集 | Web MediaDevices API | Electron 内可直接使用，减少额外服务依赖 |
| 面部特征识别 | MediaPipe Face Landmarker | 本地运行，支持人脸关键点和表情特征 |
| 情绪状态判断 | 规则型趋势引擎 | 可解释、可调参，适合 MVP 产品逻辑 |
| 小游戏框架 | Phaser.js | 适合轻量 2D 游戏，可嵌入 React/Electron |
| 本地存储 | SQLite | 桌面端可靠、可迁移、可查询 |
| 对话能力 | 规则回复 + 可选大模型 API | MVP 可离线演示，后续可增强自然语言能力 |
| 进程通信 | Electron IPC | 主进程和渲染进程标准通信方式 |

## 4. 选型分析

### 4.1 桌面应用框架

#### 候选方案

| 技术 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| Electron | 前端生态成熟；透明窗口、置顶、托盘、多窗口、IPC 支持完善；适合快速构建桌宠类产品 | 包体较大，内存占用高于原生应用 | MVP 首选 |
| Tauri | 包体小，安全模型好，性能轻 | Rust 参与度更高；复杂窗口和桌宠交互调试成本更高 | 后续产品化可评估 |
| Qt / PyQt | 原生桌面能力强，跨平台能力成熟 | UI 动画、小游戏和 Web 前端生态衔接成本较高 | 不作为 MVP 首选 |
| WPF / WinUI | Windows 桌面体验好 | 跨平台弱，前端和游戏生态不如 Web 技术栈 | 仅 Windows 产品可考虑 |

#### 选择 Electron 的原因

本产品的桌面体验重点是桌宠常驻，而不是传统表单应用。它需要：

1. 透明无边框窗口。
2. 窗口置顶。
3. 桌宠点击交互。
4. 多窗口管理，例如桌宠窗口、待办窗口、聊天窗口、小游戏窗口、设置窗口。
5. 系统托盘入口。
6. 与前端动画、Canvas 游戏、摄像头能力自然集成。

Electron 的窗口能力和 Web 技术栈天然适配这些需求。MVP 阶段更重要的是快速做出完整体验闭环，而不是优先压缩安装包体积，因此选择 Electron。

#### 实现方式

Electron 主进程负责：

1. 创建透明置顶的宠物窗口。
2. 创建普通功能面板窗口。
3. 管理系统托盘。
4. 管理 SQLite 数据库。
5. 暴露 IPC 接口给渲染进程调用。
6. 管理应用设置和窗口生命周期。

Electron 渲染进程负责：

1. 展示宠物动画。
2. 展示待办、聊天、设置、小游戏界面。
3. 调用摄像头和 MediaPipe。
4. 将识别结果发送给状态决策引擎。

#### 实现难度

中等。

主要难点在透明窗口、点击穿透、窗口拖拽、窗口置顶策略和多窗口通信。Electron 有成熟方案，但需要在体验细节上调试。

### 4.2 前端框架

#### 候选方案

| 技术 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| React + TypeScript + Vite | 组件化清晰，状态管理成熟，生态强，适合 Electron | 需要建立良好目录结构 | MVP 首选 |
| Vue + Vite | 上手快，适合表单和面板 | 团队通用性和复杂状态组织略弱于 React 生态 | 可选 |
| Next.js | 工程能力强 | SSR、路由等能力对桌面端 MVP 价值有限 | 不推荐 |
| 原生 HTML/CSS/JS | 简单直接 | 后期状态复杂后难维护 | 不推荐 |

#### 选择 React 的原因

产品内存在大量状态驱动 UI：

1. 宠物状态：idle、happy、tired、care、celebrate、sleep。
2. 用户状态：happy、calm、tired、away。
3. 任务状态：todo、doing、done、delayed、abandoned。
4. 游戏状态：ready、playing、success、finish。
5. 对话状态：输入、回复、推荐动作。

React 适合用组件和状态管理把这些状态拆清楚。TypeScript 能减少事件对象、状态字段和 IPC 参数传递中的错误。

#### 实现方式

推荐前端模块结构：

```text
src/
  app/
    PetWindow.tsx
    MainPanel.tsx
  components/
    pet/
    task/
    chat/
    game/
    settings/
  features/
    emotion/
    decision/
    pet/
    tasks/
    chat/
    games/
  shared/
    types/
    events/
    storage/
```

#### 实现难度

低到中等。

难点不在框架本身，而在状态模型和事件流设计。

### 4.3 摄像头与面部识别

#### 候选方案

| 技术 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| MediaPipe Face Landmarker | 本地运行；支持 Web；可输出人脸关键点、表情特征、是否有人脸 | 需要自己设计状态判断逻辑 | MVP 首选 |
| DeepFace / FER | 可直接输出情绪分类 | Python 依赖较重；误判体验明显；部署复杂 | 暂不采用 |
| OpenCV | 传统视觉能力强，适合人脸检测 | 表情语义能力不足，Web 集成成本较高 | 可作为后端方案备选 |
| 自训练 ONNX 模型 | 可完全定制 | 需要数据集、训练、评估和模型部署 | 后续高级版本考虑 |

#### 选择 MediaPipe Face Landmarker 的原因

MVP 阶段不应该追求“准确判断用户真实情绪”，而应该判断“可用于产品交互的状态趋势”。例如：

1. 是否检测到人脸。
2. 是否有明显微笑。
3. 眼部闭合或眨眼频率是否异常。
4. 头部姿态是否低垂。
5. 表情活跃度是否下降。

MediaPipe Face Landmarker 可以在本地浏览器环境中运行，适合 Electron。它能提供人脸关键点和表情相关特征，为产品状态判断提供输入。

#### 为什么不优先选择 DeepFace / FER

DeepFace / FER 这类方案更接近“直接输出 happy、sad、angry 等分类”。但本产品不应让用户觉得应用在武断判断自己“低落”或“焦虑”。误判一旦出现，会明显破坏陪伴感和信任感。

因此 MVP 使用更克制的产品状态：

```text
happy
calm
tired
away
```

“低落”“烦躁”“压力”不在 MVP 中直接强判断，可以在 V1.1 或 V2.0 通过更稳定的多信号融合再扩展。

#### 实现方式

```text
摄像头授权
-> 获取 video stream
-> 每 1 到 3 秒抽取一帧
-> MediaPipe Face Landmarker 推理
-> 输出 facePresent、blendshapes、landmarks
-> 特征计算模块生成 emotionSignal
-> 趋势引擎输出 userState
```

示例输入：

```json
{
  "facePresent": true,
  "smileScore": 0.72,
  "eyeClosedScore": 0.12,
  "headDownScore": 0.08,
  "activityScore": 0.64,
  "timestamp": 1710000000000
}
```

示例输出：

```json
{
  "userState": "happy",
  "confidence": 0.78,
  "duration": 42,
  "source": "face_landmarker"
}
```

#### 实现难度

中到偏高。

接入模型本身难度中等，真正难点在于阈值调优、持续时间判断、误判兜底和隐私体验。

### 4.4 情绪趋势与状态决策引擎

#### 候选方案

| 技术 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| 规则型状态机 | 可解释、可调试、成本低、适合产品策略 | 规则复杂后需要治理 | MVP 首选 |
| 机器学习分类器 | 可以学习复杂模式 | 需要标注数据和训练流程 | 不适合 MVP |
| 大模型决策 | 语义能力强 | 不适合处理摄像头状态；成本和隐私风险较高 | 不采用 |

#### 选择规则型状态机的原因

本产品的关键不是算法炫技，而是陪伴策略。比如同样检测到疲惫：

1. 如果用户刚完成任务，宠物应该庆祝后提醒休息。
2. 如果任务很多，宠物应该降低催促压力。
3. 如果刚提醒过，应该进入冷却，不要重复打扰。
4. 如果用户关闭摄像头，应该尊重选择，不再推断状态。

这些都是产品规则，使用规则型状态机更可控。

#### 状态分层

系统状态分为三层：

```text
原始识别信号 emotionSignal
-> 用户状态 userState
-> 产品动作 productAction
```

原始识别信号：

```ts
type EmotionSignal = {
  facePresent: boolean;
  smileScore: number;
  eyeClosedScore: number;
  headDownScore: number;
  activityScore: number;
  timestamp: number;
};
```

用户状态：

```ts
type UserState = "happy" | "calm" | "tired" | "away";
```

产品动作：

```ts
type ProductAction =
  | "pet_idle"
  | "pet_happy"
  | "pet_care"
  | "pet_sleep"
  | "invite_smile_game"
  | "remind_break"
  | "celebrate_task";
```

#### 规则示例

| 条件 | 用户状态 | 宠物状态 | 动作 |
| --- | --- | --- | --- |
| 30 秒内 smileScore 均值较高 | happy | happy | 增强正反馈 |
| 无明显异常且有人脸 | calm | idle | 安静陪伴 |
| eyeClosedScore 或 headDownScore 持续较高超过 30 秒 | tired | care | 关心提示 |
| tired 持续超过 3 分钟且未处于冷却 | tired | care | 邀请微笑能量小游戏 |
| 超过 2 分钟无 facePresent | away | sleep | 宠物睡觉 |
| 完成任务 | 不强依赖情绪 | celebrate | 增加亲密度 |

#### 实现难度

中等。

建议把规则配置化，避免硬编码散落在 UI 组件中。

### 4.5 桌宠动画方案

#### 候选方案

| 技术 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| WebP / PNG 序列帧 | 最稳定，设计可控，适合状态动画 | 资源体积可能偏大 | MVP 首选 |
| Lottie | 轻量，适合 UI 风格动画 | 对复杂角色动作表现有限 | MVP 可用 |
| Live2D | 表现力强，适合二次元桌宠 | 建模、授权、调参成本较高 | 后续增强 |
| Spine | 游戏级骨骼动画强 | 工具链和美术成本较高 | 后续增强 |

#### 选择 Lottie / 序列帧的原因

MVP 需要的是“状态明显、动作可爱、稳定展示”。序列帧和 Lottie 不需要复杂建模流程，可以快速实现：

1. 待机。
2. 开心。
3. 关心。
4. 疲惫。
5. 庆祝。
6. 睡觉。

#### 实现方式

宠物状态机：

```ts
type PetState = "idle" | "happy" | "care" | "tired" | "celebrate" | "sleep";
```

宠物资源映射：

```json
{
  "idle": "assets/pet/idle.webp",
  "happy": "assets/pet/happy.webp",
  "care": "assets/pet/care.webp",
  "tired": "assets/pet/tired.webp",
  "celebrate": "assets/pet/celebrate.webp",
  "sleep": "assets/pet/sleep.webp"
}
```

#### 实现难度

低到中等。

主要依赖美术资源质量。技术上不复杂，但体验好坏很大程度取决于动画节奏、气泡文案和打扰频率。

### 4.6 小游戏框架

#### 候选方案

| 技术 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| Phaser.js | 2D 游戏框架成熟；支持 Canvas/WebGL；适合 JS/TS；可嵌入 Electron | 需要学习 Phaser 场景结构 | MVP 首选 |
| 原生 Canvas | 最轻量 | 碰撞、场景、资源加载、动画都要自研 | 不推荐 |
| PixiJS | 渲染能力强，适合复杂动画 | 游戏逻辑能力没有 Phaser 完整 | 可选 |
| Unity / Godot | 游戏能力强 | 过重，和桌面宠物 Web 架构割裂 | 不适合 MVP |

#### 选择 Phaser.js 的原因

本产品的游戏是轻量 2D 游戏，单局 30 秒到 3 分钟，需要快速进入、快速结束，并和情绪状态联动。Phaser 提供场景、精灵、输入、碰撞、计时器等基础能力，可以减少手写游戏底层逻辑的成本。

#### MVP 游戏：微笑能量收集

游戏目标：

通过检测用户微笑程度，为宠物积累能量。

核心机制：

```text
用户进入小游戏
-> 游戏读取 smileScore
-> smileScore 越高，能量增长越快
-> 达到目标后宠物恢复开心状态
-> 写入游戏结果和奖励
```

情绪联动：

| 用户状态 | 游戏调整 |
| --- | --- |
| happy | 能量增长更快，奖励更多 |
| calm | 标准模式 |
| tired | 降低目标值，缩短游戏时间 |
| away | 暂停游戏，提示回到摄像头前 |

#### 实现难度

中等。

难点在于游戏和情绪识别模块之间的数据同步，以及不要让游戏因为识别延迟而显得卡顿。

### 4.7 本地存储

#### 候选方案

| 技术 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| SQLite | 成熟可靠；适合桌面端；支持查询和迁移 | 需要主进程封装数据库操作 | MVP 首选 |
| IndexedDB | 浏览器内置，无需原生依赖 | 调试、迁移、备份体验一般 | 可做轻量备选 |
| JSON 文件 | 开发最快 | 并发、查询、迁移、数据一致性较差 | 仅 Demo 可用 |
| 云数据库 | 多设备同步方便 | 隐私、登录、网络和服务端复杂度上升 | V2.0 再考虑 |

#### 选择 SQLite 的原因

本产品的数据虽然不复杂，但属于长期使用数据，包括：

1. 今日任务。
2. 历史任务完成记录。
3. 宠物亲密度。
4. 用户设置。
5. 提醒冷却记录。
6. 游戏结果。

这些数据适合结构化查询和后续迁移。SQLite 在桌面应用中成熟稳定，也方便后续做备份。

#### 数据隐私原则

默认存储：

1. 任务数据。
2. 设置数据。
3. 宠物成长数据。
4. 游戏结果。
5. 用户状态摘要，例如 tired 持续时间。

默认不存储：

1. 原始人脸图片。
2. 摄像头视频。
3. 单帧人脸截图。
4. 可识别个人身份的图像数据。

#### 实现难度

低到中等。

主要工作是数据库表设计和 IPC 数据访问封装。

### 4.8 对话模块

#### 候选方案

| 技术 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- |
| 规则回复 | 稳定、可控、可离线演示 | 自然度有限 | MVP 首选 |
| 大模型 API | 对话自然，扩展性强 | 成本、网络、隐私和安全策略复杂 | 可选增强 |
| 本地小模型 | 隐私更好 | 资源占用高，效果不稳定 | 后续探索 |

#### 选择规则回复 + 可选大模型 API 的原因

MVP 的对话目标不是做通用聊天机器人，而是让宠物根据当前状态给出陪伴式反馈。规则回复已经可以覆盖核心场景：

1. 用户说累。
2. 用户说任务多。
3. 用户想玩。
4. 用户完成任务后庆祝。
5. 用户低状态时给温和鼓励。

后续如果需要增强自然语言能力，可以增加大模型 Provider，但不影响基础闭环。

#### 实现方式

对话输入上下文：

```json
{
  "message": "我有点累",
  "userState": "tired",
  "todayTasks": 5,
  "finishedTasks": 2,
  "lastGameResult": "success",
  "petAffinity": 18
}
```

规则回复输出：

```json
{
  "reply": "那我们先慢一点吧。要不要玩 1 分钟轻松小游戏，我陪你缓一下。",
  "recommendedAction": "invite_smile_game"
}
```

#### 实现难度

低。

大模型增强为中等，主要涉及 API Key 管理、网络错误、内容安全和隐私说明。

```text
│  └──────────────┘                             │

```

### 5.2 进程职责

#### 主进程 Main Process

负责系统级能力：

1. 应用启动和生命周期。
2. 创建宠物窗口、面板窗口、设置窗口。
3. 系统托盘菜单。
4. SQLite 数据库访问。
5. 安全 IPC 接口。
6. 本地配置读写。

#### 渲染进程 Renderer Process

负责用户界面和交互：

1. React UI。
2. 桌宠展示。
3. 气泡提醒。
4. 待办面板。
5. 聊天面板。
6. 小游戏渲染。
7. 摄像头采集。
8. MediaPipe 识别。
9. 状态决策调用。

#### 为什么摄像头和 MediaPipe 放在渲染进程

因为摄像头 MediaDevices API 和 Web 版 MediaPipe 更适合在浏览器环境中运行。这样可以减少 Python 服务、Node 原生模块和多进程图像传输成本。

后续如果识别能力变复杂，可以独立为本地 AI Worker 或本地服务。

## 6. 核心模块设计

### 6.1 桌面宠物模块

#### 模块职责

1. 展示宠物动画。
2. 根据 PetState 切换动作。
3. 处理点击、拖拽、快捷菜单。
4. 展示气泡提醒。
5. 作为待办、游戏、聊天入口。

#### 状态定义

```ts
type PetState =
  | "idle"
  | "happy"
  | "care"
  | "tired"
  | "celebrate"
  | "sleep";
```

#### 事件输入

| 事件 | 来源 | 结果 |
| --- | --- | --- |
| user_state_changed | 情绪决策引擎 | 切换宠物状态 |
| task_completed | 待办模块 | 触发庆祝 |
| game_finished | 游戏模块 | 触发开心或安慰 |
| user_clicked_pet | 用户操作 | 打开快捷菜单 |
| idle_timeout | 系统计时器 | 进入待机或睡眠 |

### 6.2 情绪识别模块

#### 模块职责

1. 管理摄像头权限。
2. 控制采样频率。
3. 调用 MediaPipe 识别人脸特征。
4. 输出标准化 EmotionSignal。
5. 不保存原始图像。

#### 采样策略

MVP 建议：

1. 默认每 2 秒识别一次。
2. 游戏过程中可提升到每 500ms 到 1s 一次。
3. 用户静默模式下降低或暂停识别。
4. 窗口最小化或锁屏时暂停识别。

#### 输出对象

```ts
type EmotionSignal = {
  facePresent: boolean;
  smileScore: number;
  eyeClosedScore: number;
  headDownScore: number;
  activityScore: number;
  timestamp: number;
};
```

### 6.3 状态决策模块

#### 模块职责

1. 接收 EmotionSignal。
2. 维护滑动时间窗口。
3. 计算 UserState。
4. 根据任务、提醒冷却、用户偏好输出 ProductAction。

#### 输入

```ts
type DecisionInput = {
  signal: EmotionSignal;
  taskSummary: {
    totalToday: number;
    unfinished: number;
    overdue: number;
  };
  settings: {
    cameraEnabled: boolean;
    reminderLevel: "quiet" | "normal" | "active";
    focusMode: boolean;
  };
  lastPromptAt?: number;
};
```

#### 输出

```ts
type DecisionOutput = {
  userState: UserState;
  confidence: number;
  petState: PetState;
  action?: ProductAction;
  message?: string;
};
```

### 6.4 待办任务模块

#### 模块职责

1. 新建、编辑、删除任务。
2. 标记完成。
3. 今日任务筛选。
4. 任务奖励事件。
5. 与宠物亲密度联动。

#### 任务状态

```ts
type TaskStatus =
  | "todo"
  | "doing"
  | "done"
  | "delayed"
  | "abandoned";
```

#### 联动规则

| 用户行为 | 系统事件 | 宠物反馈 |
| --- | --- | --- |
| 新建任务 | task_created | 宠物陪伴提示 |
| 完成任务 | task_completed | 宠物庆祝，亲密度增加 |
| 任务延期 | task_delayed | 温和提醒，不惩罚 |
| 今日全部完成 | daily_tasks_done | 庆祝动画和奖励 |

### 6.5 小游戏模块

#### 模块职责

1. 管理游戏启动和结束。
2. 从情绪模块读取实时 smileScore。
3. 根据 userState 调整游戏难度。
4. 输出游戏结果和奖励。

#### MVP 游戏流程

```text
打开微笑能量收集
-> 初始化 Phaser 场景
-> 读取当前 userState
-> 设置目标能量和游戏时长
-> 实时读取 smileScore
-> 计算能量增长
-> 游戏结束
-> 写入 game_result
-> 通知宠物模块
```

### 6.6 对话陪伴模块

#### 模块职责

1. 展示聊天窗口。
2. 读取用户输入。
3. 获取情绪、任务、游戏、宠物上下文。
4. 生成规则回复。
5. 输出推荐动作。

#### 规则分类

| 用户表达 | 识别方式 | 回复策略 |
| --- | --- | --- |
| 累、困、不想动 | 关键词 + tired 状态 | 建议小游戏或短休息 |
| 烦、压力大 | 关键词 | 降低刺激，建议呼吸 |
| 任务好多 | 关键词 + unfinishedTasks | 建议拆小任务 |
| 想玩 | 关键词 | 推荐小游戏 |
| 今天做完了 | taskSummary | 庆祝和奖励 |

## 10. 隐私与安全设计

### 10.1 摄像头隐私

1. 首次使用必须明确请求摄像头授权。
2. 宠物窗口或设置页必须展示摄像头工作状态。
3. 用户可随时关闭摄像头和情绪识别。
4. 默认不保存原始图片和视频。
5. 默认不上传人脸图像。
6. 情绪记录只保存摘要状态，不保存可识别图像。

### 10.2 文案边界

产品不输出医疗判断，不使用以下表达：

1. 诊断。
2. 抑郁症。
3. 焦虑症。
4. 心理疾病。
5. 严重精神状态判断。

推荐表达：

1. “你可能有点累了。”
2. “要不要先慢一点？”
3. “我陪你休息 1 分钟。”
4. “我们先完成一个小任务。”

### 10.3 本地数据安全

1. SQLite 数据保存在用户本地应用目录。
2. 不保存摄像头原始数据。
3. 大模型 API 默认关闭，用户主动配置后才启用。
4. 如果开启大模型，只发送文本上下文，不发送人脸图像。

## 11. 性能设计

### 11.1 摄像头识别性能

1. 常规状态每 2 秒识别一次。
2. 游戏状态可提升到 0.5 到 1 秒一次。
3. 应用后台或专注模式下降低识别频率。
4. 无人脸超过 2 分钟后进入低频检测。

### 11.2 动画性能

1. 桌宠动画资源控制大小。
2. 宠物窗口只渲染必要区域。
3. 避免复杂 DOM 动画。
4. Phaser 游戏窗口不打开时不初始化游戏循环。

### 11.3 数据库性能

1. 任务、设置、游戏记录数据量小，SQLite 足够。
2. 情绪摘要不高频写入，建议按状态变化或分钟级写入。
3. 原始 EmotionSignal 只在内存滑动窗口保存。

## 12. 异常与兜底

| 异常 | 兜底策略 |
| --- | --- |
| 用户拒绝摄像头 | 产品进入普通桌宠模式，保留待办、游戏、聊天 |
| MediaPipe 加载失败 | 显示“识别暂不可用”，不影响基础功能 |
| 未检测到人脸 | 宠物进入等待或睡眠 |
| 情绪判断不确定 | 默认 calm，避免误触发 |
| SQLite 初始化失败 | 提示用户重启或重新初始化数据 |
| 大模型接口失败 | 回退规则回复 |
| 游戏中识别中断 | 使用默认能量增长，保证游戏可完成 |

## 15. 风险与应对

| 风险 | 描述 | 应对 |
| --- | --- | --- |
| 情绪识别误判 | 表情不等于真实情绪 | 使用趋势判断，不输出强情绪标签 |
| 摄像头隐私顾虑 | 用户可能抗拒持续摄像头 | 本地识别、明确开关、默认不保存图像 |
| 桌宠打扰工作 | 频繁提醒会产生反感 | 冷却时间、静默模式、低打扰气泡 |
| Electron 资源占用 | 内存和包体偏大 | 控制窗口数量、识别频率、动画资源 |
| 游戏与产品割裂 | 小游戏变成普通娱乐 | 游戏必须读取 userState，并反哺宠物成长 |
| 对话安全风险 | 用户表达强烈负面内容 | 不做医疗建议，提供温和回应和必要求助提醒 |
| 美术资源不足 | 宠物表现不够可爱 | MVP 先少状态高质量，不追求数量 |

## 17. 技术结论

MVP 最终技术方案为：

```text
Electron + React + TypeScript + Vite
MediaPipe Face Landmarker
规则型情绪趋势引擎
Phaser.js
SQLite
Lottie / WebP 序列帧
规则对话 + 可选大模型 API
```

该方案的核心优势是：

1. 能快速实现桌面宠物体验。
2. 能保护摄像头和人脸隐私。
3. 能以可解释方式完成状态判断。
4. 能低成本打通宠物、游戏、待办、对话闭环。
5. 能为后续产品化升级保留空间。

因此，本技术方案适合作为本项目 MVP 和原型展示阶段的首选方案。
