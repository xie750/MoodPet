# 情绪感知型桌面陪伴精灵产品需求文档

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 产品名称 | 情绪感知型桌面陪伴精灵 |
| 产品形态 | 桌面应用 |
| 核心能力 | 人脸情绪识别、桌面宠物、情绪驱动小游戏、待办管理、AI 对话陪伴 |
| 目标用户 | 长时间使用电脑的办公人员、学生、自习用户、远程办公用户 |
| 文档版本 | V1.0 |

## 2. 产品背景

随着办公、学习和线上娱乐逐渐转移到电脑场景中，用户长时间面对屏幕已经成为常态。传统电脑应用主要响应用户的主动操作，却无法感知用户当前的精神状态、情绪变化和疲劳程度。

对于办公人员、学生和远程工作者来说，长时间独处使用电脑容易带来疲惫、低落、压力和注意力下降等问题。普通待办软件只能提醒任务，普通小游戏只能提供短暂娱乐，普通桌面宠物只能提供视觉陪伴，它们都缺少对用户当前状态的感知能力。

本产品希望通过人脸识别和情绪识别技术，让桌面应用具备感知用户状态的能力，并通过桌面宠物、小游戏、待办任务和对话陪伴形成完整交互闭环。

## 3. 产品定位

本产品是一款面向长期电脑办公/学习用户的情绪感知型桌面陪伴应用。

产品通过摄像头无感采集用户面部状态，判断用户当前情绪趋势，并将识别结果转化为桌面宠物行为、小游戏内容、待办提醒策略和对话反馈。

一句话定位：

> 一个能够感知用户情绪，并通过桌面宠物、小游戏、待办和对话陪伴用户的智能桌面应用。

## 4. 产品核心价值

本产品的核心价值不是单纯完成“情绪识别算法”，也不是单纯开发“桌面小游戏”，而是打通以下闭环：

```text
用户状态感知
-> 情绪与疲劳趋势判断
-> 桌面宠物状态变化
-> 小游戏/待办/对话联动
-> 用户获得陪伴、鼓励和轻量放松
```

产品希望解决的问题是：

1. 电脑无法理解用户当前状态。
2. 用户长时间办公或学习时缺少低打扰陪伴。
3. 普通待办工具容易产生压力，缺少情绪化反馈。
4. 普通小游戏与用户状态无关，缺少主动干预价值。
5. 普通桌宠只有装饰作用，缺少智能感知和实际场景结合。

产品差异化价值：

1. 让桌面宠物不只是可爱，而是能根据用户状态做出反应。
2. 让小游戏不只是娱乐，而是根据用户情绪变化调整内容。
3. 让待办任务不只是提醒，而是结合情绪提供更温和的陪伴式管理。
4. 让对话不只是聊天，而是结合用户状态、任务和宠物关系进行反馈。

## 5. 目标用户与使用场景

### 5.1 目标用户

| 用户类型 | 用户特征 | 核心需求 |
| --- | --- | --- |
| 办公人员 | 长时间面对电脑，容易疲惫、焦虑、任务堆积 | 放松提醒、任务陪伴、情绪缓冲 |
| 学生群体 | 自习、备考、写作业时长期独处 | 专注陪伴、任务激励、情绪反馈 |
| 远程办公用户 | 工作环境较孤独，缺少实时互动 | 桌面陪伴、轻量互动、情绪支持 |
| 桌宠爱好者 | 喜欢桌面美化、宠物养成、轻度游戏 | 可爱宠物、互动反馈、成长系统 |
| 效率工具用户 | 需要待办、提醒、日程管理 | 每日任务、完成奖励、低压力管理 |

### 5.2 典型场景

场景一：疲惫提醒

用户连续办公 1 小时，系统检测到用户表情活跃度下降、闭眼频率升高或疲惫状态持续。桌面宠物变成担心状态，发出轻提醒：“感觉你坚持很久啦，要不要休息 1 分钟？”

场景二：情绪驱动小游戏

用户当前状态被判断为低落或疲惫，宠物主动邀请用户进入 30 秒小游戏。小游戏自动切换为低压力模式，降低难度，增加鼓励反馈。

场景三：任务完成反馈

用户完成今日待办任务后，宠物庆祝并增加亲密度。任务完成奖励可以用于解锁宠物装饰、小游戏道具或桌面动作。

场景四：对话陪伴

用户点击宠物并输入“我今天有点烦”。宠物结合当前情绪状态和待办情况，给出简短、温和、不说教的回应，并建议用户先完成一个小任务或玩一局放松小游戏。

## 6. 产品功能架构

产品由五个主要模块组成：

```text
情绪识别模块
桌面宠物模块
情绪驱动小游戏模块
待办任务模块
对话陪伴模块
```

其中，情绪识别模块是感知层，桌面宠物是交互层，小游戏是干预层，待办任务是办公场景层，对话陪伴是情感表达层。

## 7. 功能需求

### 7.1 人脸情绪识别模块

#### 7.1.1 功能说明

系统通过摄像头采集用户面部信息，识别人脸状态和基础情绪，并输出当前用户状态标签。

该模块重点不在于医疗级情绪判断，而在于为后续宠物、游戏、待办和对话提供触发依据。

#### 7.1.2 识别状态

MVP 阶段建议先支持以下状态：

| 状态 | 含义 | 产品用途 |
| --- | --- | --- |
| 开心 | 用户表情积极，状态较好 | 增强宠物互动，提升游戏奖励 |
| 平静 | 用户状态稳定，无明显负面情绪 | 保持普通陪伴状态 |
| 疲惫 | 用户可能长时间工作或注意力下降 | 触发休息提醒或低强度小游戏 |
| 低落 | 用户表情消极，活跃度较低 | 触发安慰、陪伴或鼓励内容 |
| 烦躁/压力 | 用户可能皱眉、紧张或不耐烦 | 降低提醒频率，引导呼吸/放松 |
| 离开 | 未检测到人脸 | 宠物进入等待或睡眠状态 |

#### 7.1.3 状态判断逻辑

系统不应基于单帧画面立即判断用户情绪，而应基于一段时间内的趋势进行判断。

建议规则：

| 条件 | 输出状态 | 触发动作 |
| --- | --- | --- |
| 连续 30 秒检测到疲惫特征 | 轻度疲惫 | 宠物变为关心状态 |
| 连续 3 分钟疲惫或低落 | 需要关怀 | 触发小游戏邀请 |
| 连续 60 分钟在电脑前 | 长时间专注 | 触发休息提醒 |
| 完成任务后检测到积极表情 | 积极反馈 | 增强奖励动画 |
| 用户不在摄像头前超过 2 分钟 | 离开状态 | 宠物等待或睡觉 |

#### 7.1.4 输出数据

情绪识别模块向系统其他模块输出标准状态对象：

```json
{
  "emotion": "tired",
  "confidence": 0.76,
  "duration": 180,
  "suggestedAction": "invite_relax_game"
}
```

#### 7.1.5 隐私要求

1. 摄像头必须由用户主动授权。
2. 系统应明确显示摄像头工作状态。
3. 优先采用本地识别方案。
4. 默认不保存原始人脸图片或视频。
5. 不将人脸图像上传到服务器。
6. 用户可随时关闭情绪识别。
7. 产品文案避免使用“诊断”“抑郁症”“焦虑症”等医疗表达。

### 7.2 桌面宠物模块

#### 7.2.1 功能说明

桌面宠物是产品的核心视觉载体和主要交互入口。用户可以选择自己喜欢的宠物，宠物常驻桌面，并根据用户情绪、任务状态和游戏结果产生不同表现。

#### 7.2.2 核心功能

1. 宠物选择：用户可从多个宠物形象中选择一个作为桌面伙伴。
2. 桌面常驻：宠物可停留在桌面角落或窗口边缘。
3. 状态动画：宠物支持待机、开心、担心、疲惫、庆祝、睡眠等状态。
4. 点击交互：用户点击宠物后打开快捷菜单。
5. 气泡提示：宠物可用低打扰气泡发出提醒。
6. 模块入口：宠物可作为小游戏、待办、聊天入口。
7. 养成反馈：任务完成和游戏结果可提升亲密度或解锁装饰。

#### 7.2.3 宠物状态映射

| 用户状态/行为 | 宠物表现 | 产品目的 |
| --- | --- | --- |
| 用户开心 | 宠物跳跃、挥手 | 强化积极反馈 |
| 用户平静 | 宠物安静陪伴 | 保持低打扰 |
| 用户疲惫 | 宠物递茶、打哈欠、提醒休息 | 提供轻量关怀 |
| 用户低落 | 宠物靠近、安慰、邀请小游戏 | 建立陪伴感 |
| 用户烦躁 | 宠物降低动效，提示深呼吸 | 避免刺激用户 |
| 用户离开 | 宠物睡觉或等待 | 表现真实陪伴 |
| 完成任务 | 宠物庆祝、亲密度增加 | 提升成就感 |

#### 7.2.4 宠物快捷菜单

用户点击宠物后出现快捷菜单：

| 菜单项 | 功能 |
| --- | --- |
| 今日任务 | 打开待办任务面板 |
| 陪我玩 | 打开小游戏列表 |
| 和我聊天 | 打开对话窗口 |
| 换宠物 | 进入宠物选择 |
| 设置 | 摄像头、提醒频率、隐私设置 |

### 7.3 情绪驱动小游戏模块

#### 7.3.1 功能说明

产品内置 3 到 5 个轻量小游戏。小游戏不仅作为娱乐内容，还应根据用户当前情绪调整玩法、难度、画面和反馈方式。

核心特色：

> 用户情绪能够改变游戏内容。

#### 7.3.2 游戏设计原则

1. 单局时长短，建议 30 秒到 3 分钟。
2. 操作简单，适合办公间隙快速进入。
3. 难度可根据用户状态动态调整。
4. 游戏结果可反哺宠物成长和待办奖励。
5. 尽量避免高挫败感设计。

#### 7.3.3 推荐小游戏

##### 游戏一：微笑能量收集

玩法：

用户通过微笑为宠物积累能量。摄像头检测到微笑时，游戏中的能量球增加，宠物逐渐恢复活力。

情绪联动：

| 用户状态 | 游戏表现 |
| --- | --- |
| 开心 | 能量增长更快，奖励更多 |
| 平静 | 标准模式 |
| 疲惫 | 降低目标数值，缩短游戏时间 |
| 低落 | 增加鼓励文案和柔和画面 |
| 烦躁 | 切换为慢节奏，不强调分数 |

该游戏适合作为 MVP 首个游戏，因为它能直接体现人脸识别与游戏内容的结合。

##### 游戏二：情绪跑酷

玩法：

用户控制宠物躲避障碍、收集能量道具。游戏节奏根据用户状态变化。

情绪联动：

| 用户状态 | 游戏变化 |
| --- | --- |
| 开心 | 游戏速度略快，连击奖励增加 |
| 疲惫 | 降低速度，减少障碍 |
| 低落 | 增加鼓励道具 |
| 烦躁 | 切换低压力模式，减少失败惩罚 |

##### 游戏三：呼吸节奏

玩法：

当系统检测到用户烦躁或压力较高时，宠物引导用户完成 30 到 60 秒的呼吸节奏小游戏。

情绪联动：

1. 压力状态越高，节奏越慢。
2. 游戏不设置失败，强调完成与放松。
3. 完成后宠物给予温和鼓励。

##### 游戏四：任务花园

玩法：

用户完成待办任务后获得阳光、水滴或种子，用于让花园成长。宠物会在花园中活动。

情绪联动：

1. 用户状态稳定时，花园更加明亮。
2. 用户低落时，宠物邀请用户收集阳光。
3. 完成任务越多，花园成长越明显。

##### 游戏五：桌面接水果

玩法：

用户控制宠物接住掉落物品。游戏时长短，适合办公间隙放松。

情绪联动：

| 用户状态 | 游戏变化 |
| --- | --- |
| 疲惫 | 掉落速度降低 |
| 开心 | 连击奖励增加 |
| 低落 | 奖励文案更温柔 |
| 烦躁 | 去掉失败提示，保留轻松反馈 |

#### 7.3.4 游戏触发方式

| 触发来源 | 示例 |
| --- | --- |
| 用户主动点击 | 用户点击宠物菜单中的“陪我玩” |
| 情绪触发 | 用户疲惫持续 3 分钟，宠物邀请小游戏 |
| 任务触发 | 完成任务后解锁奖励小游戏 |
| 对话触发 | 用户说“我有点累”，宠物推荐放松小游戏 |

### 7.4 待办任务模块

#### 7.4.1 功能说明

待办任务模块用于帮助用户管理每日任务，并与宠物成长、情绪状态和小游戏奖励进行联动。

该模块不应只做普通任务列表，而应强调“陪伴式任务管理”。

#### 7.4.2 基础功能

1. 新建任务。
2. 编辑任务。
3. 删除任务。
4. 标记完成。
5. 设置截止时间。
6. 设置优先级。
7. 今日任务列表。
8. 已完成任务统计。

#### 7.4.3 任务状态

| 状态 | 含义 |
| --- | --- |
| 未开始 | 用户已创建但尚未处理 |
| 进行中 | 用户正在处理 |
| 已完成 | 用户完成任务 |
| 已延期 | 超过截止时间 |
| 已放弃 | 用户主动取消 |

#### 7.4.4 待办与宠物联动

| 用户行为 | 宠物反馈 |
| --- | --- |
| 创建任务 | 宠物表示“我陪你一起完成” |
| 临近截止 | 宠物进行轻提醒 |
| 完成任务 | 宠物庆祝并增加亲密度 |
| 连续完成任务 | 解锁装饰或小游戏奖励 |
| 任务堆积 | 宠物建议拆分任务 |
| 用户疲惫且任务未完成 | 宠物建议先休息再继续 |
| 用户低落 | 宠物减少催促，改用鼓励式提醒 |

#### 7.4.5 待办与小游戏联动

| 条件 | 奖励 |
| --- | --- |
| 完成 1 个任务 | 获得宠物能量 |
| 完成 3 个任务 | 解锁小游戏奖励局 |
| 连续专注 25 分钟 | 开启休息小游戏 |
| 完成今日全部任务 | 宠物庆祝动画 |
| 任务延期 | 宠物安慰并建议重新规划 |

### 7.5 对话陪伴模块

#### 7.5.1 功能说明

用户可以与桌面宠物进行对话。宠物根据用户当前情绪、今日任务、游戏结果和历史偏好给出反馈。

对话模块应服务于陪伴、鼓励、任务辅助和游戏引导，不应被设计成医疗心理咨询。

#### 7.5.2 对话场景

| 场景 | 示例 |
| --- | --- |
| 主动聊天 | 用户想和宠物说话 |
| 情绪表达 | 用户说“我有点累” |
| 任务询问 | 用户问“今天还有什么任务” |
| 游戏推荐 | 用户说“想放松一下” |
| 完成反馈 | 用户完成任务后宠物庆祝 |
| 状态关心 | 宠物主动发出轻问候 |

#### 7.5.3 对话风格

宠物对话应满足以下原则：

1. 简短。
2. 温和。
3. 不说教。
4. 不诊断。
5. 不制造压力。
6. 更像伙伴，而不是老师或医生。

示例：

```text
用户：我有点累。
宠物：那我们先慢一点吧。要不要玩 1 分钟轻松小游戏，我陪你缓一下。
```

```text
用户：今天任务好多。
宠物：我们先挑一个最小的开始。完成一个也算前进，我陪你。
```

```text
用户：我不想做了。
宠物：可以先停一下。你不是失败了，只是需要换口气。
```

#### 7.5.4 上下文输入

对话模块应能读取以下信息：

1. 当前情绪状态。
2. 今日任务数量。
3. 已完成任务数量。
4. 最近一次小游戏结果。
5. 宠物亲密度。
6. 用户偏好设置。

## 8. 核心用户流程

### 8.1 首次使用流程

```text
打开应用
-> 选择宠物
-> 授权摄像头
-> 设置隐私偏好
-> 创建今日任务
-> 宠物出现在桌面
-> 开始陪伴
```

### 8.2 情绪触发小游戏流程

```text
用户长时间办公
-> 系统检测到疲惫/低落状态
-> 状态持续超过阈值
-> 宠物发出轻提示
-> 用户接受小游戏邀请
-> 游戏根据情绪切换模式
-> 游戏结束
-> 宠物给予鼓励
-> 返回桌面陪伴状态
```

### 8.3 待办任务流程

```text
用户创建任务
-> 宠物进入陪伴状态
-> 到达提醒时间
-> 宠物发出低打扰提醒
-> 用户完成任务
-> 宠物庆祝
-> 增加亲密度或游戏奖励
```

### 8.4 对话流程

```text
用户点击宠物
-> 打开对话窗口
-> 用户输入内容
-> 系统结合情绪和任务上下文
-> 宠物回复
-> 可进一步推荐小游戏、休息或任务拆分
```

## 9. MVP 版本规划

### 9.1 MVP 目标

MVP 版本重点验证完整闭环，而不是追求功能数量。

验证链路：

```text
情绪识别
-> 宠物状态变化
-> 宠物主动邀请
-> 情绪驱动小游戏
-> 待办奖励
-> 对话反馈
```

### 9.2 MVP 必须实现

| 优先级 | 功能 | 说明 |
| --- | --- | --- |

### 9.3 MVP 推荐演示流程

```text
用户打开应用并选择宠物
-> 摄像头开始识别用户状态
-> 用户状态为疲惫
-> 宠物变成关心状态
-> 宠物邀请用户玩“微笑能量收集”
-> 用户通过微笑给宠物充能
-> 游戏结束后宠物恢复开心
-> 用户完成一个待办任务
-> 宠物庆祝并增加亲密度
```

## 11. 技术方案建议

### 11.1 推荐技术架构

## 12. 非功能需求

### 12.1 体验要求

1. 宠物必须可爱、美观，有明显情绪表现。
2. 应用不能过度打扰用户。
3. 提醒应使用轻气泡，而不是强弹窗。
4. 游戏应短平快，不影响办公节奏。
5. 待办模块应避免制造压力。

### 12.2 性能要求

1. 摄像头识别不应明显拖慢电脑。
2. 宠物动画应保持流畅。
3. 情绪识别可降低采样频率，例如每 1 到 3 秒识别一次。
4. 小游戏启动时间应尽量短。

### 12.3 隐私要求

1. 本地优先。
2. 明确授权。
3. 可随时关闭。
4. 不默认保存人脸图像。
5. 不上传原始视频。
6. 不输出医疗诊断结论。

## 13. 风险与应对

| 风险 | 描述 | 应对策略 |
| --- | --- | --- |
| 情绪识别误判 | 表情不一定等于真实情绪 | 使用趋势判断，并允许用户反馈 |
| 用户隐私顾虑 | 摄像头采集容易引起不信任 | 本地识别、明确提示、可关闭 |
| 桌宠打扰办公 | 频繁提醒会影响用户 | 设置提醒等级和静默模式 |
| 游戏与办公割裂 | 小游戏可能变成独立娱乐 | 通过情绪和任务奖励强绑定 |
| 功能过多导致开发困难 | 桌宠、算法、游戏、待办、聊天都要做 | MVP 先做一个完整闭环 |
| 对话安全风险 | 用户可能表达强烈负面情绪 | 避免医疗建议，必要时给出求助提醒 |

## 14. 产品边界

本产品不是医疗诊断工具，不判断用户是否患有心理疾病。

本产品不是单纯的桌面宠物，也不是单纯的待办软件或小游戏合集。

本产品的边界是：

1. 识别用户状态趋势。
2. 提供低打扰陪伴。
3. 通过游戏和任务反馈缓解疲惫。
4. 帮助用户更温和地完成日常任务。

## 15. 项目亮点总结

本项目的创新点可以总结为：

1. 情绪识别驱动桌面交互。
2. 桌面宠物作为情绪反馈载体。
3. 小游戏内容根据用户状态动态变化。
4. 待办任务与宠物成长深度结合。
5. 对话陪伴结合情绪与任务上下文。
6. 面向办公/学习场景，而不是单纯娱乐。

最终产品表达：

> 情绪识别是感知层，桌面宠物是陪伴层，小游戏是干预层，待办任务是场景层，对话系统是情感表达层。

## 17. 结论

该产品的核心不是“给小游戏加一个情绪识别算法”，而是建立一个完整的情绪感知桌面陪伴系统。

产品最有价值的闭环是：

```text
用户情绪被感知
-> 宠物理解并做出反应
-> 游戏和待办根据状态调整
-> 用户获得陪伴、鼓励和轻量放松
```

因此，本产品应重点突出：

1. 用户状态可以改变宠物表现。
2. 用户情绪可以改变游戏内容。
3. 用户任务可以推动宠物成长。
4. 用户对话可以获得上下文陪伴。

这使产品区别于普通桌宠、普通待办、普通小游戏和普通聊天工具，形成具有展示价值和产品潜力的创新方向。
