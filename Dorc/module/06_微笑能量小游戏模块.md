# 微笑能量小游戏模块设计文档

## 1. 模块定位

微笑能量小游戏是 MVP 最重要的差异化展示模块。

它要证明产品不是普通桌宠，也不是普通小游戏，而是“用户状态能改变游戏内容”。

## 2. 游戏概念

游戏名称：微笑能量收集

玩法：

```text
用户进入游戏
-> 宠物能量较低
-> 用户微笑时 smileScore 上升
-> 游戏能量条增长
-> 能量满后宠物恢复开心
```

## 3. MVP 目标

1. 使用 Phaser.js 或轻量 Canvas 实现游戏场景。
2. 读取实时 smileScore。
3. smileScore 越高，能量增长越快。
4. 根据 UserState 调整目标值。
5. 游戏结束后写入本地记录。
6. 发布 `game_finished` 事件，由应用编排层决定宠物反馈。

如果项目已经接入 Phaser，优先使用 Phaser。没有接入前，可以先用 React + Canvas 做临时版本，但最终建议 Phaser。

## 4. 游戏状态

```ts
type SmileGameState =
  | "ready"
  | "playing"
  | "paused"
  | "success"
  | "finished"
  | "quit";
```

## 5. 难度规则

| UserState | 游戏调整 |
| --- | --- |
| happy | 能量增长更快，奖励更多 |
| calm | 标准目标 |
| tired | 降低目标值，缩短时长 |
| away | 暂停，提示回到摄像头前 |

MVP 参数建议：

```ts
const configByState = {
  happy: { targetEnergy: 80, maxSeconds: 45, reward: 2 },
  calm: { targetEnergy: 100, maxSeconds: 60, reward: 1 },
  tired: { targetEnergy: 60, maxSeconds: 45, reward: 1 },
  away: { targetEnergy: 100, maxSeconds: 60, reward: 0 },
};
```

## 6. 能量计算

建议公式：

```text
energyGainPerTick = baseGain + smileScore * smileMultiplier
```

规则：

1. smileScore 高时能明显加速。
2. smileScore 低时也缓慢增长，避免用户尴尬。
3. 游戏不应该失败，只区分完成、普通结束和退出。

## 7. 依赖接口

小游戏模块不直接依赖情绪识别模块实现，只依赖共享接口：

```ts
type EmotionSignalReader = {
  getLatestSignal(): EmotionSignal | null;
  subscribe(listener: (signal: EmotionSignal) => void): () => void;
};

type GameEventPublisher = {
  publish(event: GameFinishedEvent): void;
};
```

这样小游戏可以在真实摄像头、模拟数据或测试数据下运行。

## 8. 游戏结束输出

```ts
type GameRecord = {
  id: string;
  gameType: "smile_energy";
  result: "success" | "normal" | "quit";
  score: number;
  duration: number;
  reward: string;
  createdAt: number;
};
```

游戏模块只输出记录和事件，不直接修改宠物状态。

## 9. 反馈文案

成功：

```text
能量回来啦，谢谢你陪我充电。
```

普通结束：

```text
这样也很好，我们慢慢来。
```

退出：

```text
没关系，想玩的时候我还在。
```

## 10. 验收标准

1. 可以从宠物菜单进入游戏。
2. 游戏能读取 smileScore。
3. 微笑时能量增长更快。
4. tired 状态下游戏目标更低。
5. away 状态下游戏暂停或温和提示。
6. 游戏结束后写入 game_records。
7. 游戏结束后发布 game_finished 事件。
8. 游戏模块不直接调用宠物模块。
