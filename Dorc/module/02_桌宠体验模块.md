# 桌宠体验模块设计文档

## 1. 模块定位

桌宠是产品的核心体验载体。

用户对产品的第一感知不是“情绪识别算法”，而是这个宠物是否可爱、是否懂分寸、是否能在合适时机给出陪伴反馈。

## 2. MVP 目标

MVP 阶段只做一个宠物，但要把体验做完整：

1. 宠物常驻桌面。
2. 支持 6 个状态。
3. 支持点击交互。
4. 支持快捷菜单。
5. 支持低打扰气泡。
6. 支持任务完成和游戏结束后的反馈。

## 3. 宠物状态

```ts
type PetState =
  | "idle"
  | "happy"
  | "care"
  | "tired"
  | "celebrate"
  | "sleep";
```

| 状态 | 含义 | 典型表现 |
| --- | --- | --- |
| idle | 默认陪伴 | 安静呼吸、轻微摆动 |
| happy | 积极反馈 | 挥手、跳跃、笑 |
| care | 关心用户 | 递茶、靠近、轻提示 |
| tired | 陪用户慢下来 | 打哈欠、放慢动作 |
| celebrate | 任务/游戏奖励 | 撒花、鼓掌 |
| sleep | 用户离开 | 睡觉、等待 |

## 4. 状态来源

桌宠模块不直接消费任务、游戏、情绪识别模块的原始事件。

桌宠只接收应用编排层或状态决策引擎生成的 `PetCommand`：

```ts
type PetCommand = {
  type: "set_state" | "show_bubble" | "add_affinity" | "add_energy";
  state?: PetState;
  message?: string;
  amount?: number;
  reason?: string;
};
```

推荐映射由状态决策引擎或应用编排层完成：

| 上游事件 | 编排后的 PetCommand |
| --- | --- |
| user_state_changed: happy | set_state: happy |
| user_state_changed: calm | set_state: idle |
| user_state_changed: tired | set_state: care |
| user_state_changed: away | set_state: sleep |
| task_completed | set_state: celebrate + add_affinity |
| game_finished: success | set_state: happy + add_energy |
| game_finished: quit | set_state: care |

## 5. 快捷菜单

点击宠物后显示菜单：

| 菜单项 | 行为 |
| --- | --- |
| 今日任务 | 打开主面板的任务页 |
| 陪我玩 | 打开微笑能量小游戏 |
| 和我聊天 | 打开聊天页 |
| 设置 | 打开设置页 |

MVP 不做“换宠物”，因为当前只做一个宠物。后续 V1.1 再加入。

## 6. 气泡提醒原则

气泡提醒是低打扰，不是弹窗。

原则：

1. 一次最多显示 1 条。
2. 自动消失，建议 4 到 6 秒。
3. 不遮挡用户主要工作区。
4. 同类提醒必须有冷却时间。
5. 文案短，不说教，不诊断。

示例：

```text
感觉你坚持很久啦，要不要缓 1 分钟？
我在这儿，我们先慢一点。
完成一个啦，真不错。
```

## 8. 模块边界

桌宠模块可以做：

1. 渲染宠物状态。
2. 展示气泡。
3. 处理点击菜单。
4. 执行 `PetCommand`。

桌宠模块不可以做：

1. 直接读取摄像头识别结果。
2. 直接查询任务列表来决定提醒。
3. 直接写入游戏记录。
4. 直接决定疲惫、开心、离开等用户状态。
5. 直接实现提醒冷却策略。

## 9. MVP 资源策略

如果没有正式美术资源，先使用占位资源，但状态必须可区分。

优先级：

1. idle。
2. care。
3. happy。
4. celebrate。
5. sleep。
6. tired。

可以使用 WebP、PNG 序列帧、Lottie 或 CSS 动画占位。

## 10. 验收标准

1. 宠物有明显状态切换。
2. 点击宠物能打开菜单。
3. 气泡不会无限重复出现。
4. 完成任务后宠物能庆祝。
5. 用户离开后宠物进入 sleep。
6. 状态变化不依赖具体 UI 页面。
7. 除 `PetCommand` 外，不接受其他模块直接修改宠物状态。
