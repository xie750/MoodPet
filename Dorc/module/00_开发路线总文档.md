# 情绪感知型桌面陪伴精灵开发路线总文档

## 1. 对 vibe coding 思路的修正

如果把 vibe coding 理解成“想到哪里写到哪里”，这个方向需要修正。

本项目涉及桌面窗口、宠物状态、摄像头、情绪识别、小游戏、待办、对话和本地存储。如果没有模块边界，后续让 AI 写代码时很容易出现状态混乱、功能割裂、重复实现和难以调试的问题。

更适合本项目的 vibe coding 方式是：

```text
先定义可演示闭环
-> 拆成小模块文档
-> 每次只让 AI 实现一个模块或一个用户流程
-> 保持模块输入输出稳定
-> 跑通后再扩展体验
```

也就是说，本项目不是先追求“大而全”，而是先做出一个能打动人的最小产品闭环。

## 2. 产品开发主线

本产品的第一目标不是做一个复杂工具，而是验证：

```text
用户状态被感知
-> 桌宠产生可信反应
-> 用户被温和引导
-> 小游戏/待办/对话形成正反馈
```

MVP 演示主链路：

```text
启动应用
-> 桌宠出现
-> 用户授权摄像头
-> 系统识别 happy / calm / tired / away
-> 桌宠根据状态切换表现
-> 疲惫时邀请微笑能量小游戏
-> 游戏结束后宠物恢复活力
-> 用户完成一个今日任务
-> 宠物庆祝并增加亲密度
-> 用户可通过对话获得简短陪伴反馈
```

## 4. 模块协作原则

为避免多人分模块开发时互相影响，所有模块必须遵守以下原则：

1. 业务模块之间不直接互相调用。
2. 跨模块协作只能通过共享类型、事件总线或 `window.appApi` 暴露的接口完成。
3. 桌宠状态只有桌宠模块可以渲染和持有运行态。
4. 情绪识别模块只输出 `EmotionSignal`，不直接决定宠物动作。
5. 状态决策引擎只输出 `DecisionOutput`，不直接操作 UI 和数据库。
6. 待办模块只管理任务数据和任务事件，不直接修改宠物状态。
7. 小游戏模块只管理游戏流程和游戏事件，不直接修改宠物状态。
8. 设置模块只管理用户偏好，不直接控制其他模块内部实现。
9. 本地存储模块只负责持久化，不承载产品规则。

统一调用方向：

```text
模块内部能力
-> 发布领域事件
-> 状态决策/应用编排层消费事件
-> 生成 PetCommand / NavigationCommand / StorageCommand
-> 对应模块执行
```

应用编排层不是新的业务模块，而是应用层胶水代码，建议统一放在：

```text
src/app/orchestrator/
```

它只做事件订阅、命令分发和模块连接，不写具体业务算法，不渲染 UI，不定义数据库表。

并行开发前必须先阅读：

```text
00_开发路线总文档.md
11_模块接口契约与事件总线.md
当前要开发的模块文档
```

## 5. 推荐开发顺序

### 阶段一：先让桌宠“活着”

目标：用户打开应用后能看到一个稳定、可点击、可拖动、会切状态的桌宠。

开发模块：

1. `01_应用壳与窗口模块.md`
2. `02_桌宠体验模块.md`
3. `10_视觉资源与文案模块.md`

验收结果：

```text
桌宠透明置顶显示
-> 可拖动
-> 点击出现快捷菜单
-> 可手动切换 idle / happy / care / sleep / celebrate
-> 可展示低打扰气泡
```

### 阶段二：打通状态决策

目标：先不用真实摄像头，也能用模拟数据驱动桌宠变化。

开发模块：

1. `11_模块接口契约与事件总线.md`
2. `03_状态决策引擎模块.md`
3. `09_本地存储与IPC模块.md`

验收结果：

```text
输入模拟用户状态 tired
-> 决策引擎输出 petState = care
-> 桌宠展示关心状态
-> 气泡显示温和提醒
-> 冷却期内不重复打扰
```

### 阶段三：接入待办闭环

目标：任务完成能触发宠物庆祝和亲密度增长。

开发模块：

1. `05_待办任务模块.md`
2. `09_本地存储与IPC模块.md`
3. `02_桌宠体验模块.md`

验收结果：

```text
创建今日任务
-> 完成任务
-> 写入本地数据库
-> 宠物切换 celebrate
-> 亲密度增加
```

### 阶段四：接入情绪识别

目标：摄像头本地识别用户基础状态，驱动桌宠。

开发模块：

1. `04_情绪识别模块.md`
2. `03_状态决策引擎模块.md`
3. `08_设置与隐私模块.md`

验收结果：

```text
用户授权摄像头
-> 本地识别 facePresent / smileScore / eyeClosedScore
-> 输出 happy / calm / tired / away
-> 桌宠自动切状态
```

### 阶段五：实现微笑能量小游戏

目标：完成最关键的“情绪改变游戏内容”演示。

开发模块：

1. `06_微笑能量小游戏模块.md`
2. `04_情绪识别模块.md`
3. `03_状态决策引擎模块.md`
4. `09_本地存储与IPC模块.md`

验收结果：

```text
用户进入小游戏
-> smileScore 越高能量增长越快
-> tired 状态下降低目标值
-> 游戏结束写入记录
-> 宠物恢复 happy 或 care
```

### 阶段六：补齐对话与设置

目标：让产品看起来像完整 MVP，而不是技术 Demo。

开发模块：

1. `07_对话陪伴模块.md`
2. `08_设置与隐私模块.md`

验收结果：

```text
用户输入“我有点累”
-> 宠物结合 tired 状态和未完成任务
-> 回复短句
-> 推荐休息或微笑能量小游戏
```

## 6. 每次让 AI 开发时的推荐输入方式

不要一次性说“帮我做完整项目”。推荐说：

```text
请根据 module/01_应用壳与窗口模块.md，
并遵守 module/11_模块接口契约与事件总线.md，
先实现 Electron + React + TypeScript + Vite 的应用壳，
只完成宠物透明窗口、主面板窗口和基础 IPC，
不要实现情绪识别、小游戏和待办。
```

每次开发只给 AI 一个模块文档和必要的关联文档。这样代码更集中，也更容易验收。

## 7. MVP 必须坚持的边界

MVP 要做：

1. 单个宠物。
2. 4 个用户状态：happy、calm、tired、away。
3. 6 个宠物状态：idle、happy、care、tired、celebrate、sleep。
4. 一个小游戏：微笑能量收集。
5. 今日待办最小功能。
6. 规则对话。
7. 本地存储。
8. 摄像头可关闭，默认不保存图像。

MVP 不做：

1. 多宠物商城。
2. 情绪周报。
3. 医疗级情绪判断。
4. 复杂心理咨询。
5. 五个小游戏。
6. 云同步。
7. Live2D 高级建模。
8. 完整日历系统。

## 8. 产品经理验收标准

最终 MVP 不以“功能数量”验收，而以“闭环是否成立”验收：

1. 用户是否能明显感知宠物陪伴？
2. 宠物状态变化是否合理、不过度打扰？
3. 情绪识别误判时是否仍然不冒犯用户？
4. 小游戏是否真的被用户状态影响？
5. 待办完成是否带来情绪正反馈？
6. 对话是否短、温和、不说教？
7. 隐私开关是否明确可信？

只要这条闭环成立，后续扩展才有意义。

## 9. 本次模块冲突审查结论

原模块拆分存在以下潜在冲突，已在各模块文档和 `11_模块接口契约与事件总线.md` 中修正：

1. 情绪识别模块和状态决策引擎都可能输出 `UserState`。
   修正：情绪识别只输出 `EmotionSignal`，`UserState` 只由状态决策引擎计算。

2. 待办模块、小游戏模块可能直接修改桌宠状态。
   修正：待办和小游戏只发布事件，桌宠只执行 `PetCommand`。

3. 多个模块都可能直接增加宠物亲密度或能量。
   修正：亲密度和能量只通过宠物命令处理方写入。

4. 对话模块可能直接打开页面或启动小游戏。
   修正：对话只返回推荐动作，用户确认后由应用壳执行导航命令。

5. 设置模块可能直接控制摄像头、窗口和提醒策略。
   修正：设置模块只保存设置并发布 `settings_changed`，其他模块自行响应。

6. 本地存储模块可能承载产品规则。
   修正：本地存储只做持久化，不判断是否庆祝、提醒或邀请游戏。

7. 应用壳和本地存储都可能定义数据库初始化。
   修正：应用壳只调用本地存储初始化，不定义表结构。
