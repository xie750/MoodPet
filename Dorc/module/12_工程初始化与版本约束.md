# 工程初始化与版本约束

## 1. 文档目的

本文件用于约束正式工程的初始化方式、依赖版本、目录结构和开发规范。

后续所有代码实现必须优先遵守：

```text
00_开发路线总文档.md
11_模块接口契约与事件总线.md
12_工程初始化与版本约束.md
```

如果模块文档与本文件冲突，以本文件为准。

## 2. 首发平台与工程目标

MVP 首发平台：

```text
Windows 10 / Windows 11
```

MVP 工程目标：

1. 快速跑通桌宠、待办、情绪识别、小游戏、对话和设置闭环。
2. 保持 Electron 主进程、preload、渲染进程职责清晰。
3. 优先保证隐私、安全、可调试和可打包。
4. 不为了早期演示牺牲模块边界。

MVP 暂不追求：

1. 自动更新。
2. 多平台安装包。
3. 云同步。
4. 插件系统。
5. 高复杂度 Live2D / Spine 工作流。

## 3. 技术栈版本约束

推荐版本：

| 类型 | 约束 |
| --- | --- |
| Node.js | `>=20.18.0 <23` |
| npm | `>=10` |
| Electron | `^31` 或 `^32` |
| React | `^18.3` |
| TypeScript | `^5.5` |
| Vite | `^5.4` |
| Phaser | `^3.80` |
| SQLite 驱动 | 优先 `better-sqlite3`，如打包受阻再评估 `sqlite3` |
| MediaPipe | `@mediapipe/tasks-vision` |
| 动画 | MVP 使用 WebP / PNG / Lottie，暂不引入 Live2D |

包管理器：

```text
npm
```

MVP 阶段不混用 npm、pnpm、yarn。仓库内只保留一个 lock 文件：

```text
package-lock.json
```

## 4. TypeScript 约束

必须启用：

```json
{
  "strict": true,
  "noUncheckedIndexedAccess": true,
  "noImplicitOverride": true
}
```

类型定义原则：

1. 跨模块共享类型统一放在 `src/shared/types/`。
2. IPC 入参和出参必须有明确类型。
3. 数据库实体类型与 UI ViewModel 类型分开。
4. 不在 React 组件中临时声明跨模块类型。
5. 不使用 `any` 穿透模块边界。

允许局部使用 `unknown`，但必须在边界处完成校验和收窄。

## 5. 正式工程目录结构

正式工程统一放在：

```text
app/
```

推荐结构：

```text
app/
  package.json
  tsconfig.json
  vite.config.ts
  electron/
    main.ts
    preload.ts
    windows/
    ipc/
    db/
  src/
    app/
      PetWindow.tsx
      MainPanel.tsx
      orchestrator/
    components/
      pet/
      task/
      chat/
      game/
      settings/
    features/
      emotion/
      decision/
      pet/
      tasks/
      chat/
      games/
    shared/
      types/
      events/
      constants/
      utils/
  assets/
    pet/
    icons/
    sounds/
  public/
    mediapipe/
```

禁止在 `app/` 下随意新增平级业务目录。新增目录前必须确认它属于：

1. Electron 主进程能力。
2. React 渲染进程能力。
3. 共享类型或工具。
4. 静态资源。

## 6. 进程边界

### 6.1 Main Process

主进程可以做：

1. 创建窗口。
2. 管理托盘。
3. 管理应用生命周期。
4. 访问 SQLite。
5. 注册 IPC handler。
6. 读取应用路径和资源路径。

主进程不可以做：

1. 渲染 React UI。
2. 执行情绪识别算法。
3. 直接写业务 UI 状态。
4. 保存原始摄像头图像。

### 6.2 Preload

preload 只做安全桥接：

1. 使用 `contextBridge.exposeInMainWorld` 暴露 `window.appApi`。
2. 每个 API 只调用白名单 IPC channel。
3. 不暴露 `ipcRenderer` 原始对象。
4. 不暴露 Node.js `fs`、`path`、`process` 等对象。

### 6.3 Renderer Process

渲染进程可以做：

1. React UI。
2. 桌宠动画。
3. 摄像头采集。
4. MediaPipe 推理。
5. Phaser 游戏。
6. 调用 `window.appApi`。

渲染进程不可以做：

1. 直接访问 SQLite。
2. 直接访问本地文件系统。
3. 直接调用 Electron 主进程内部模块。
4. 自行拼接任意 IPC channel。

## 7. 代码风格约束

命名规则：

| 对象 | 规则 | 示例 |
| --- | --- | --- |
| React 组件 | PascalCase | `PetWindow` |
| Hook | camelCase + use | `usePetState` |
| 类型 | PascalCase | `EmotionSignal` |
| 常量 | UPPER_SNAKE_CASE | `DEFAULT_REMINDER_COOLDOWN_MS` |
| 文件夹 | kebab-case 或单词小写 | `smile-energy` |
| IPC channel | `domain:action` | `tasks:create` |

代码组织：

1. UI 组件不直接写业务规则。
2. 业务规则不直接操作 DOM。
3. 数据库层不承载产品策略。
4. 事件总线只传递领域事件，不传递 React 组件实例。
5. 同一个状态只能有一个写入方。

## 8. 依赖引入约束

新增依赖前必须满足至少一个条件：

1. 明显降低复杂度。
2. 是领域核心能力，例如 Phaser、MediaPipe、SQLite。
3. 替代手写易错逻辑，例如 schema 校验。
4. 已经被工程架构预留。

MVP 阶段暂不引入：

1. Redux。
2. 大型 UI 组件库。
3. 服务端框架。
4. 云数据库 SDK。
5. 埋点分析 SDK。
6. 自动更新框架。

## 9. 环境变量约束

环境变量文件：

```text
app/.env
app/.env.local
```

MVP 默认不需要云端密钥。

如后续启用大模型 API，必须使用：

```text
VITE_LLM_PROVIDER_ENABLED=false
VITE_LLM_PROVIDER_NAME=
```

禁止把 API Key 写入：

1. Git 仓库。
2. 前端源码。
3. 默认配置文件。
4. 示例测试数据。

## 10. 开发命令约束

推荐命令：

```text
npm run dev
npm run build
npm run typecheck
npm run lint
```

开发期必须保证：

1. `npm run typecheck` 通过。
2. 关键业务规则有单元测试或 mock 验证。
3. 主进程启动失败时有明确日志。
4. 摄像头不可用时应用仍能进入普通桌宠模式。

## 11. 正式开工前验收

正式工程初始化完成后必须满足：

1. `app/` 目录存在。
2. 主进程、preload、渲染进程目录已分开。
3. `window.appApi` 类型已声明。
4. 共享类型已包含 `EmotionSignal`、`UserState`、`PetState`、`AppEvent`。
5. IPC channel 常量集中定义。
6. SQLite schema 和 migration 目录已预留。
7. assets 目录已预留宠物资源位置。
8. PoC 代码与正式工程代码没有互相 import。
