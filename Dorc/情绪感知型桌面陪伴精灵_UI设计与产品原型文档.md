## 2. 产品经理视角下的 UI 定位

本产品不是普通效率工具，也不是纯娱乐桌宠。UI 的核心目标不是堆功能，而是让用户在长时间办公、学习、自习时感受到：

```text
它在陪我
它懂得克制
它尊重我的隐私
它能在我疲惫时轻轻拉我一下
```

因此 UI 设计必须同时满足三条主线：

1. 陪伴感：宠物是主角，界面服务于宠物与用户之间的互动。
2. 低打扰：提醒、气泡、小游戏邀请都应轻，不应强弹窗、强打断。
3. 隐私可信：摄像头和情绪识别相关 UI 必须明确、可关闭、可理解。

## 3. 已确认的产品设计决策

### 3.2 宠物形象方向

采用 A 方向：

```text
小云朵 / 小团子风
```

设计理由：

1. 情绪陪伴产品需要安全、柔软、不刺激的视觉感受。
2. 云朵/团子形象不会强绑定性别、年龄、动物偏好。
3. 形态简单，适合 MVP 用 Lottie、WebP 序列帧或 CSS 动画快速实现。
4. 表情和动作可控，能清楚表达开心、关心、睡眠、庆祝等状态。

形象关键词：

```text
柔软、圆润、轻盈、温暖、可亲近、有一点点灵性
```

避免方向：

```text
医疗感、儿童玩具感、强二次元、强科技机器人、过度拟人化
```

### 3.3 MVP 原型定位

MVP 原型允许并建议“两者结合”：

1. 展示汇报型：能让观看者快速看懂产品亮点和闭环。
2. 开发落地型：每个界面都有明确布局、状态、交互和验收标准。

因此本 UI 文档不会只写漂亮概念，也不会只写工程控件。每个关键界面都需要描述：

```text
页面目标
入口
布局
关键组件
交互规则
状态变化
文案示例
开发验收点
```

### 3.4 情绪结果展示原则

UI 不直接展示强情绪判断结果。

不建议出现：

```text
你现在低落
你现在烦躁
你压力很大
系统判断你情绪异常
```

建议显示为“陪伴模式”和自然语言提示：

| 系统内部状态 | 面向用户展示 |
| --- | --- |
| happy | 活力陪伴中 |
| calm | 安静陪伴中 |
| tired | 关心一下 |
| away | 等你回来 |

示例：

```text
安静陪伴中
小精灵会保持低打扰，需要时再轻轻提醒你。
```

```text
关心一下
你可能坚持有一会儿了，要不要休息 1 分钟？
```

这样既能体现产品智能，又避免用户觉得被评判。

### 3.5 主面板形态

主面板采用：

```text
轻量桌面控制中心
```

不是全屏后台，不是复杂管理台，也不是移动端大卡片页面。

推荐形态：

```text
默认窗口：760 x 560
最小窗口：640 x 480
紧凑模式：420 x 560
圆角：12px
布局：左侧窄导航 + 右侧内容区
```

产品感受：

```text
像一个温和的小控制台，打开即可快速看今天状态、任务、游戏和聊天。
```

主面板必须支持个性化：

1. 浅色 / 深色主题。
2. 紧凑 / 舒展布局密度。
3. 宠物大小。
4. 气泡提醒强度。
5. 是否开机启动。
6. 桌宠停靠位置。
7. 陪伴语气偏好。

## 7. 核心用户流程

### 7.1 首次使用流程

```text
打开应用
-> 欢迎页
-> 选择小云朵 / 小团子宠物
-> 隐私说明
-> 摄像头授权
-> 选择提醒强度
-> 创建第一个今日任务
-> 桌宠出现
-> 主面板最小化
```

设计目标：

1. 不让用户一开始就被设置项淹没。
2. 先建立陪伴感，再解释识别能力。
3. 摄像头授权必须解释“本地识别、不保存图像”。

### 7.2 日常陪伴流程

```text
用户工作 / 学习
-> 桌宠保持 idle 或 calm 动画
-> 用户点击宠物
-> 出现快捷菜单
-> 进入任务 / 游戏 / 聊天 / 设置
```

设计目标：

1. 用户不操作时，宠物不能抢注意力。
2. 宠物动作应细微循环，避免长时间大幅跳动。
3. 桌宠是入口，但不是阻碍。

### 7.3 疲惫关怀流程

```text
系统内部判断 tired
-> 宠物切换 care 状态
-> 显示轻气泡
-> 用户可选择“休息 1 分钟”或“稍后”
-> 接受后进入微笑能量小游戏
-> 游戏结束后宠物恢复 happy / idle
```

用户看到的文案：

```text
你可能坚持有一会儿了。
要不要和我一起休息 1 分钟？
```

不显示：

```text
检测到你处于疲惫状态
疲惫置信度 0.76
```

### 7.4 任务完成反馈流程

```text
用户完成任务
-> 任务项勾选完成
-> 宠物切换 celebrate
-> 气泡反馈
-> 亲密度 +1 / 能量 +5
-> 首页数据刷新
```

反馈文案：

```text
完成一个就很棒啦，我记下来啦。
```

### 7.5 聊天陪伴流程

```text
用户点击“和我聊天”
-> 打开聊天面板
-> 用户输入“我有点累”
-> 宠物结合当前陪伴模式与任务上下文回复
-> 展示推荐动作按钮
```

推荐动作按钮示例：

```text
休息 1 分钟
玩微笑能量
看看今日任务
```

## 8. 视觉设计系统

### 8.1 设计风格

推荐风格：

```text
Soft Minimal Companion UI
```

中文解释：

```text
柔和极简陪伴式界面
```

设计要求：

1. 界面干净，信息密度适中。
2. 背景柔和，不用强烈大渐变。
3. 组件圆润，但不要全是大圆药丸。
4. 颜色不能单一，主色、辅助色、奖励色要有清晰分工。
5. 深色模式要安静，不使用高饱和霓虹。

4. 深色模式不是简单反色，需要单独配置文本、边框和表面色。


字号层级：

| 用途 | font-weight |
| --- | --- |

| 区块标题 | 600 |

### 8.4 圆角与阴影

| Token | 值 | 用途 |
| --- | --- | --- |
| radius-xs | 6px | 标签、小按钮 |
| radius-sm | 8px | 输入框、任务项 |
| radius-md | 10px | 弹层、气泡 |
| radius-lg | 12px | 主面板 |
| radius-full | 999px | 开关、进度胶囊 |

注意：

1. 普通卡片圆角不超过 8px。
2. 主面板作为窗口容器可以 12px。
3. 不要大量使用 24px 以上圆角，避免幼稚感。

### 8.5 间距规范

采用 4px / 8px 基准：

| Token | 值 |
| --- | --- |
| space-1 | 4px |
| space-2 | 8px |
| space-3 | 12px |
| space-4 | 16px |
| space-5 | 20px |
| space-6 | 24px |
| space-8 | 32px |
| space-10 | 40px |

布局规则：

1. 主面板内边距 20px。
2. 页面区块之间 20px 到 24px。
3. 列表项内部 12px 到 16px。
4. 小按钮点击区域不小于 32px，高频按钮不小于 36px。
5. 桌面端主要按钮高度 36px 到 40px。

## 9. 组件规范

### 9.1 按钮

按钮类型：

| 类型 | 用途 | 样式 |
| --- | --- | --- |
| Primary | 当前页面主操作 | 主色背景 + 白字 |
| Secondary | 次级动作 | 浅色底 + 主色文字 |
| Ghost | 弱操作 | 透明底，hover 有浅色层 |
| Danger | 删除、关闭危险能力 | danger 色或 danger 文字 |
| IconButton | 关闭、更多、拖动辅助 | 图标居中 |

尺寸：

| 尺寸 | 高度 | 横向内边距 |
| --- | --- | --- |
| sm | 32px | 12px |
| md | 36px | 14px |
| lg | 40px | 18px |

状态：

1. hover：背景加深 4% 到 8%。
2. pressed：轻微缩放到 0.98，仅按钮自身，不影响布局。
3. disabled：透明度 0.45，禁用点击。
4. loading：保留原宽度，显示小型加载指示。

### 9.2 输入框

输入框用于：

1. 新建任务。
2. 编辑任务。
3. 聊天输入。

样式：

```text
高度：36px / 聊天输入 42px
圆角：8px
边框：1px solid border.subtle
背景：bg.panel
focus：2px focus 外描边
```

错误提示：

1. 显示在输入框下方。
2. 文案说明如何修复。
3. 不只显示“错误”。

### 9.3 开关

用于：

1. 摄像头开启。
2. 情绪识别开启。
3. 静默模式。
4. 开机启动。

规则：

1. 开关右侧必须有文字标题。
2. 下方可有一句解释。
3. 涉及隐私的开关，关闭后立即生效。

### 9.4 分段控件

用于：

1. 浅色 / 深色 / 跟随系统。
2. 紧凑 / 标准 / 舒展。
3. 安静 / 标准 / 主动提醒。

样式：

```text
高度：36px
圆角：8px
选中态：主色浅底 + 主色文字
```

### 9.5 任务项

任务项结构：

```text
[完成圆点] 任务标题
          截止时间 / 优先级 / 宠物提示
          [编辑] [删除]
```

状态表现：

| 状态 | 表现 |
| --- | --- |
| 未开始 | 默认文字 |
| 进行中 | 左侧细主色条 |
| 已完成 | 文字弱化 + 完成动效 |
| 已延期 | 显示温和提示，不大面积红色 |
| 已放弃 | 弱化展示，可恢复 |

延期文案：

```text
这个任务晚了一点，可以重新安排。
```

避免文案：

```text
你逾期了
任务失败
```

### 9.6 气泡

气泡是桌宠最关键的陪伴表达。

尺寸：

```text
最小宽度：180px
最大宽度：280px
内边距：12px 14px
圆角：10px
最多显示：2 行主文案 + 1 行操作
```

位置：

1. 默认出现在宠物上方或左上方。
2. 避免遮挡桌面主要操作区域。
3. 如果靠近屏幕边缘，自动反向展开。

类型：

| 类型 | 触发 | 示例 |
| --- | --- | --- |
| 轻问候 | 启动、空闲 | 我在这儿，慢慢来。 |
| 关心 | tired | 要不要休息 1 分钟？ |
| 任务反馈 | 完成任务 | 完成一个就很棒啦。 |
| 游戏邀请 | tired 持续 | 玩一局微笑能量？ |
| 隐私提示 | 摄像头状态 | 识别已暂停。 |

气泡规则：

1. 非用户主动触发的气泡自动消失，默认 6 秒。
2. 关心类气泡 15 分钟内最多出现 1 次。
3. 静默模式下只显示任务完成反馈，不主动邀请游戏。
4. 气泡必须可手动关闭。

不做：

1. 多宠物商城。
2. 装饰道具商城。
3. 复杂换装。

### 开发验收

1. 修改昵称后全局生效。
2. 修改大小后桌宠窗口更新。
3. 修改主题后主面板和弹层更新。
4. 设置持久化。

## 11. 动效设计

### 11.1 动效原则

1. 动效表达因果，不做纯装饰。
2. 单次动效 150ms 到 300ms。
3. 宠物状态动画可以循环，但幅度必须小。
4. 支持 reduced motion，降低或关闭非必要动效。
5. 不动画宽高，优先使用 transform 和 opacity。

### 11.2 关键动效

| 场景 | 动效 |
| --- | --- |
| 宠物 idle | 轻微上下浮动 2px 到 4px |
| 宠物 happy | 小跳 1 次，随后回到 idle |
| 宠物 care | 宠物向气泡方向轻靠近 |
| 宠物 sleep | 眼睛闭合，呼吸幅度降低 |
| 气泡出现 | opacity 0 -> 1，translateY 6px -> 0 |
| 菜单出现 | scale 0.98 -> 1，opacity 0 -> 1 |
| 任务完成 | 勾选 + 任务项轻微淡出到已完成区 |
| 游戏能量 | 粒子飞向能量条或宠物 |

## 13. 可访问性与可用性要求

1. 所有按钮支持键盘访问。
2. 图标按钮必须有 aria-label。
3. 文本对比度满足 WCAG AA。
4. 不只用颜色表达状态，需配合文字或图标。
5. 动效支持 reduced motion。
6. 输入框有可见 label 或明确上下文标题。
7. 删除、重置等危险操作必须二次确认。
8. Toast 不抢焦点。
9. 主面板关闭后，桌宠仍可作为入口。

## 15. MVP UI 验收标准

### 15.1 闭环验收

必须跑通：

```text
首次启动
-> 选择小云朵宠物
-> 说明隐私并授权摄像头
-> 桌宠出现在桌面
-> 主面板展示今日陪伴模式
-> 创建并完成任务
-> 宠物庆祝
-> 进入微笑能量小游戏
-> 游戏结束后宠物恢复活力
-> 聊天中推荐休息或小游戏
```

### 15.2 体验验收

1. 用户不会在界面上看到被强判断的情绪标签。
2. 用户能随时看见摄像头和识别开关。
3. 桌宠不会频繁弹窗打扰。
4. 任务完成有明显正反馈。
5. 小游戏能体现状态联动。
6. 深色 / 浅色切换视觉完整。
7. 主面板不像后台管理系统，保持轻量陪伴感。

### 15.3 开发验收

1. 每个 P0 页面可进入。
2. 每个 P0 页面有空状态、错误状态、加载状态。
3. 所有设置可持久化。
4. UI 状态与业务状态解耦。
5. 隐私设置关闭后相关 UI 立即更新。
6. 文案不触碰医疗诊断边界。

## 17. 总结

本 UI 方案的核心判断是：

```text
桌宠是主角
主面板是轻量控制中心
小游戏是情绪联动展示点
待办是办公学习场景入口
聊天是陪伴表达出口
设置与隐私是信任基础
```

MVP 阶段不追求功能数量，而追求一个清晰、可信、温和的闭环：

```text
用户被陪伴
-> 状态被温和感知
-> 宠物给出低打扰反馈
-> 用户通过任务或小游戏获得正反馈
-> 产品建立继续使用的理由
```
